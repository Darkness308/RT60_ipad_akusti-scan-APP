02 Scope — und warum er geschnitten wurde (EKS × Pareto)

Die ehrliche Antwort auf „ist der Scope zu groß?": Ja. Per EKS (ein Engpass) und Pareto (20 % Aufwand → 80 % Wert, ohne Widerspruch hier ausführbar) wurde getrennt, was jetzt Substanz bringt und was bewusst der macOS-Migration gehört.

✓ In-Scope — erledigt & hier verifizierbar

  • DIN-Engine normtreu (A1–A5, Bild-2-Band) CI
  • Report ohne erfundene Werte (M3) CI
  • Volumen-Gültigkeit (H2) & 8000-Hz-Kennzeichnung (H1) CI
  • Daten-Überschreibung verhindert (2× HIGH) PR
  • Schein-Tests → echte Asserts CI
  • Ehrliches RT60-Labeling CI
  • Governance: echtes CODEOWNERS; LICENSE; Onboarding; HANDOFF §10/§11

◷ Out-of-Scope — dokumentiert für macOS/Owner

  • Echter Audio-Messpfad (Mikrofon + ISO 3382 + Schroeder) — großes Feature, gerätegebunden
  • App-Test-Target ins CI-Gate (.pbxproj-Chirurgie, nur auf Mac sicher)
  • Lint/Format-, Release/Archive-, Coverage-Gates in CI
  • ExportView verdrahten + asymmetrische Toleranz im App-Exporter
  • Modell-Duplikate App↔Package konsolidieren
  • 4000-Hz-Bild-2-Toleranz gegen gedruckte Norm prüfen (nicht raten)
Engpass-Prinzip (EKS): Der eine Engpass ist „App-Tests laufen in keiner CI". Er ist aber Mac-gebunden (diese Linux-Sandbox hat kein swift/xcodebuild). Ein Gate hier „blind" einzubauen würde die Regel „kein Green-Washing" verletzen — deshalb dokumentiert statt vorgetäuscht.

04 Datenfluss — wie eine Zahl in den Report kommt

Der reale Pfad heute. Wichtig für die Wahrheit: Der RT60-Wert entsteht aus Geometrie × Materialabsorption (Sabine), nicht aus einer akustischen Messung.

Raum-GeometrieRoomPlan/LiDAR oder manuell — nur Volumen/Fläche
→
Flächen + MaterialSurfaceStore · MaterialManager (α je Oktave)
→
RT60 = 0.161·V/ASabine — Prognose, keine Messung
↓
DIN-BewertungT_soll(A1–A5) + Bild-2-Band → within/zu hoch/zu tief
→
ReportModel+ Validity-Flags (Volumen-Range, nicht bewertete Bänder)
→
PDF / HTMLechte T_soll · ehrliches Methodik-Label

Vorher (gefährlich)

8000 Hz fiel lautlos aus der Bewertung · out-of-range-Volumen lieferte still extrapolierten Sollwert · Report druckte erfundene 0.6/0.5/0.48 · „Messung durchgeführt".

Nachher (ehrlich)

8000 Hz „nicht bewertet n. DIN" · Volumen-Range im Report markiert · echte din_targets · „RT60 berechnet (Sabine), keine Messung".

Schutz (neu)

Gescanntes Volumen wird nicht durch manuelle Defaults überschrieben · canPersist-Guard + .corrupt-Backup gegen stillen Datenverlust.

09 CI, Governance & Systemdurchsetzung

Was die CI wirklich tut

  • ci-honest.yml triggert auto (push/PR) — keine Fehler-Maskierung
  • Job 1: swift test für Consolidated + Export
  • Job 2: xcodebuild build der App (Simulator-SDK, ohne test)
  • Bei Fehler: echter Build-Tail als PR-Kommentar
  • Alle Masking-Workflows (self-healing, auto-retry, …) stillgelegt

Durchsetzung — nur der Owner kann das

  • Branch-Protection auf main + die 2 ci-honest-Jobs als required
  • „Require review from Code Owners" (CODEOWNERS jetzt real)
  • Fork-Actions: read-only + Approval für Outside-Collaborators
Status dieser drei Schalter ist aus der Sandbox nicht prüfbar — im GitHub-UI verifizieren (HANDOFF §11).

Verifizierungs-Abdeckung (qualitativ)

Engine (RT60/DIN/Impulsantwort)
Report-Rendering (PDF/HTML)
App-Logik (Store/Material/XLSX) — Tests existieren, laufen aber nicht
Audio-Messung / LiDAR / Persistenz auf Gerät

10 Fork-Reife & macOS-Migration

Ehrlicher Reifegrad für die Übergabe an einen externen Entwickler — ohne Peinlichkeiten, weil transparent.

Steht für den Fork

  • Rechenkern substanziell, normtreu, getestet, CI-grün
  • Keine Falschbehauptungen mehr in Report/App
  • Daten-Integrität: reale Clobber-Bugs behoben
  • Konsistente Doku: README · CONTRIBUTING · HANDOFF · CLAUDE.md · LICENSE · ONBOARDING
  • Security-GO (keine Secrets/Netzwerk/Telemetrie)

Erste Schritte auf dem Mac

  1. swift test (Consolidated + Export) → echtes Rot/Grün
  2. xcodebuild build der App (Simulator)
  3. App-Test-Target in Xcode anlegen → xcodebuild test ins CI (Engpass schließen)
  4. Branch-Protection + required checks aktivieren (HANDOFF §11)
  5. Entscheidung Critical: echten Audio-Messpfad bauen oder dauerhaft als „Prognose" labeln
Leitregel (in CLAUDE.md verankert): Das Wort „verifiziert" nur verwenden, wenn wirklich gebaut/gelaufen — nicht für „sieht gut aus".
